# ロードマップ

Phase 1（MVP）以降の機能追加計画です。各 Phase の完了は前 Phase の受入基準が
満たされていることを前提とします。スケジュールは暫定で、実装着手時に見直します。

---

## Phase 2 — 品質保証 & 外部連携

**狙い**: エージェントの成果物をユーザーの手作業なしで検証でき、GitHub に流し込めるようにする。

### 2.1 自動検証ループ

- タスク完了前に worktree 内で `build` / `test` / `lint` を自動実行
- 失敗時はログを Copilot セッションにフィードバックし、最大 N 回（既定 3）リトライ
- 検証コマンドはリポジトリごとに設定可能（プリセット: Node / Python / .NET / Go）
- UI では「Verification」タブで各ステップの結果を表示

### 2.2 GitHub 連携（すべて**ユーザー明示操作**がトリガ）

**方針**: アプリが自発的に `git push` / PR 作成 / Issue クローズを行うことは Phase 2 以降も行わない。すべてユーザー明示操作（ボタン押下）がトリガで、既定で無効。

- リポジトリを GitHub のリモートと関連付け、**タスクメニューから「Push this branch」を明示的に選んだときのみ** push を実行
- 同じく「Create Pull Request...」メニューから、ユーザーが PR 本文を確認した上で作成（下書き生成は自動、**作成は手動承認**）
  - PR 本文の下書きにエージェントのサマリと diff ハイライトを埋め込む
- Issue 同期: GitHub Issue をタスクとしてインポートする機能を提供。**完了時の自動クローズは既定で無効**、オプションで opt-in

### 2.3 チャットインターフェース

- Kanban に並ぶ「Codebase Chat」タブ
- リポジトリに対して自由質問できる（タスク化せず、単発セッション）

---

## Phase 3 — 知識と最適化

**狙い**: 過去タスクから学び、より良い計画・実装を行えるようにする。

### 3.1 マルチセッションメモリ

- 完了タスクのサマリ、採用アーキテクチャ判断、失敗事例を
  エンベディング付きで SQLite + ベクター拡張（`sqlite-vss`）に保存
- 新規タスク投入時に類似過去タスクを検索し、プロンプトへ自動注入

### 3.2 自動コンフリクト解消

- `merge` 時に発生した conflict を専用セッションに渡し、解消案を提示
- ユーザーは提示された解消案を UI 上で承認／拒否

### 3.3 ロードマップ / スペックビュー

- 大きな目標を複数タスクに自動分解
- 依存グラフ表示と、完了順序の自動提案

### 3.4 プロファイリングと最適化

- 長時間走るセッションのメモリ使用量計測
- Copilot の model 自動選択（簡単なタスクには軽量モデル）
